# 趋势对比 精细化修改规划

> 基于现有 `Compare.vue`、`NavTrendChart.vue`、`TrendChart.vue` 与后端 `products.py` 的现状，制定的分阶段修改规划。

---

## 一、现状概览

### 1.1 已有能力

| 模块 | 功能 |
|------|------|
| **Compare.vue** | 产品选择（持仓快速选择 + 手动代码输入）、时间范围（7/30/90/180天）、图表类型切换（归一化净值 / 七日年化）、最新值汇总表 |
| **NavTrendChart** | 单产品净值/年化、多产品归一化（基准=100）、现金管理类自动识别并双轴展示、Tooltip 显示原始净值 |
| **TrendChart** | 多产品七日年化折线图 |
| **API** | `GET /api/products/compare?codes=...&days=N` 返回按日期聚合的历史点 |

### 1.2 数据字段（HistoryPoint）

- `as_of_date`、`annualized_7d`、`income_per_10k`、`cumulative_nav`、`daily_growth_rate`

### 1.3 已知限制与痛点

- 产品选择仅展示代码/名称，无风险等级、封闭期、七日年化等摘要
- 时间范围固定 4 档，无自定义
- 图表交互：无数据点点击、无导出、无全屏
- 混合对比时现金管理类与净值型使用不同 Y 轴，认知负担
- 数据表只显示最新一条，无区间收益统计、无排序
- 后端 compare 使用 `date('now')` 而非各产品最新日期对齐窗口，节假日可能缺数据

## 二、精细化修改方向（按优先级）

### P0：核心体验

1. **对齐数据窗口**  
   - 后端 compare 以「各产品最新可用日期」为锚点，向前取 N 天，减少节假日带来的空白区间  
   - 单产品 history 已实现，compare 需同步逻辑

2. **数据表增强**  
   - 增加「区间收益率」「区间年化」列（起止日期净值差）  
   - 支持按七日年化 / 区间年化 / 累计净值排序  
   - 显示风险等级、封闭期（若已解析）

3. **产品选择体验**  
   - 持仓产品展示：名称 + 七日年化 + 风险等级（若有）  
   - 支持搜索过滤（名称/代码）  
   - 选中态更明显，支持拖拽调整顺序

### P1：图表与交互

4. **图表交互**  
   - 点击图例显隐对应曲线  
   - 可选：数据点 hover 高亮、十字辅助线  
   - 可选：导出为 PNG

5. **时间范围扩展**  
   - 增加 1 年、自定义日期范围（起止日期选择）  
   - 后端 `days` 参数扩展或新增 `from_date/to_date`

6. **混合产品提示优化**  
   - 现金管理类 vs 净值型混选时，图例/提示更清晰区分  
   - 可选：允许用户勾选「仅对比同类产品」以减少误解

### P2：进阶能力

7. **基准线**  
   - 可选：在图中增加水平基准线（如 0%、2%、3%）  
   - 可选：以某产品为基准，其他产品相对其的偏离度

8. **表格导出**  
   - 导出对比数据表为 CSV / Excel

9. **保存对比组合**  
   - 将常用产品组合保存为「收藏」，下次一键加载  
   - 需本地存储或后端接口（若有多设备需求）

## 四、技术要点与风险

### 4.1 后端 compare 时间窗口对齐

当前 SQL 使用 `date('now', '-' || :days || ' days')`，假期无净值时会少几天。

建议改为：

```sql
-- 伪代码：每个产品取 MAX(as_of_date)，再取 MIN(各产品 max_date) 作为参考
-- 窗口 = [reference_date - days, reference_date]
```

单产品 history 已有类似逻辑，可复用 `latest AS (SELECT MAX(as_of_date)...)` 思路。

### 4.2 区间收益率计算

- 净值型：`(end_nav / start_nav - 1) * 100`  
- 区间年化：`(end_nav / start_nav)^(365/actual_days) - 1`，再乘以 100  
- 现金管理类：若只有七日年化，可标注「无区间净值」，或近似用万份收益累积（复杂，可选）

### 4.3 类型与数据兼容

- `HistoryPoint` 已包含 `cumulative_nav`、`daily_growth_rate`，前端可计算区间指标  
- 需处理 `history` 为空或单点情况，避免除零

## 六、封闭期限改进（已实施）

针对「期限很多都没有、有的也不准确、分类筛选困难」，已实现：

- **product_lockup_periods 表**：存储手动设置的封闭期限，source='manual'，任何同步脚本不得覆盖
- **PUT/DELETE /api/lockup-periods/{code}**：手动设置/清除封闭期限
- **优先级**：手动设置 > 名称解析 > 未定义
- **验证机制**：展示来源标识「手动」（绿色）/「解析」（琥珀色），未定义显示「+」可点击添加
- **高级排名**：封闭期列可点击编辑，预设 日开/7天/14天/30天/90天/180天/清除
- **筛选**：增加「九十天」「未定义期限」选项
